# MEES 文档规范

本文是仓库贡献者快速指南。MkDocs 发布范围内的规范依据见 [文档与编号规范](docs/00_Introduction/02_文档与编号规范.md)；两处规则发生差异时，以发布版规范为准。

## 命名

- 目录使用两位数字前缀保持稳定排序。
- 中文主题文件使用清晰名称，避免“最终版”“最新版”等词。
- 模板文件以 `_Template` 或“模板”结尾。

## 文档编号

- `MEES-LIF-NNN`：生命周期、过程架构和跨域总览，例如 `MEES-LIF-001`。
- `MEES-PRO-0NN`：跨域总控过程，例如项目、需求、验证、配置和变更管理。
- `MEES-PRO-DNN`：专业过程，`D` 对应能力域序号的首个有效数字，例如产品 `1xx`、系统 `3xx`、软件 `4xx`、测试 `5xx`。
- `MEES-REV-VNN-NNN`：版本基线评审记录，例如 `MEES-REV-V02-001`。
- 编号在仓库内必须唯一；新增、拆分或废止文档时不得复用既有编号。

## 结构

每篇正式过程文档采用统一章节模板。概念类文档可简化，但必须包含目的、范围、核心内容和关联文档。

## Markdown

- 标题只使用一个 H1。
- 表格保持简洁，复杂流程优先使用 Mermaid。
- 内部链接使用相对路径。
- 标准内容以原创解读和工程映射为主，不复制受版权保护原文。

## 状态

- 草稿：内容未完成。
- 评审中：已具备完整结构，等待评审。
- 已批准：可以作为组织规范使用。
- 已废止：保留历史，但不得用于新项目。

模拟角色签署的演练基线使用“已批准（模拟审计）”，不得表述为真实独立批准、评估或认证。
