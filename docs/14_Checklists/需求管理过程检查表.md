# 需求管理过程检查表

## 1. 基础完整性

- [ ] 需求来源、需求类型和需求层级已定义。
- [ ] 需求治理覆盖规则、登记、状态、跨层评审、基线批准、变更和追溯。
- [ ] 需求与架构、验证、发布和配置管理接口明确。

## 2. 输入输出

- [ ] 输入包含客户需求、市场需求、法规约束、安全约束、网络安全约束和项目范围。
- [ ] 输出只包含治理规则、跨层索引、评审记录、基线批准包、追溯矩阵和变更影响台账。
- [ ] 系统需求规格和软件需求规格分别引用系统工程和软件工程的受控输出。

## 3. 角色职责

- [ ] 产品负责人确认业务目标、优先级和范围。
- [ ] 需求工程负责人只对治理规则、跨层状态、追溯、评审和基线批准包最终负责。
- [ ] 系统负责人和软件负责人分别对系统需求规格和软件需求规格最终负责。
- [ ] 系统、软件、测试、质量、安全和网络安全角色参与相关需求评审。

## 4. 工作产品

- [ ] 跨层需求索引包含受控规格引用、版本、所有者、状态和基线归属。
- [ ] 专业需求规格的内容检查由系统工程和软件工程检查表承担。
- [ ] 需求追溯矩阵连接上游需求、设计元素、测试用例和缺陷。
- [ ] 需求变更记录包含原因、影响分析、批准和实施状态。

## 5. 评审与批准

- [ ] 正式需求经过清晰性、完整性、一致性、可行性和可验证性检查。
- [ ] 需求基线经过产品、系统、软件、测试和质量代表批准。
- [ ] 安全和网络安全需求由对应领域负责人确认。

## 6. 配置与变更

- [ ] 需求治理规则、索引、基线批准包、追溯矩阵和评审记录纳入配置管理。
- [ ] 需求变更触发对架构、测试、发布和风险的影响分析。
- [ ] 需求状态与项目计划和发布范围保持一致。

## 7. 度量指标

- [ ] 需求评审通过率可统计。
- [ ] 需求变更率可统计。
- [ ] 需求覆盖率和验证完成率可追踪。
- [ ] 未关闭需求问题数可见。

## 8. 实施证据

- [ ] 需求治理产物有版本记录，专业需求规格有受控引用。
- [ ] 需求评审问题有关闭证据。
- [ ] 需求基线记录可追溯。
- [ ] 需求变更有影响分析和批准记录。

## 9. ASPICE / ISO 33020 关注点

- [ ] 支撑 SYS.2 系统需求分析。
- [ ] 支撑 SWE.1 软件需求分析。
- [ ] 覆盖 PA1.1、PA2.1 和 PA2.2。
- [ ] 为 PA3.1 过程定义提供需求管理规则基础。

## 10. 裁剪规则

- [ ] 概念验证项目裁剪后仍保留需求来源、验收准则和变更记录。
- [ ] 客户交付、安全相关或量产项目未裁剪追溯和基线管理。
- [ ] 裁剪结论有质量负责人确认。
