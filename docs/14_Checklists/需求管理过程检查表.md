# 需求管理过程检查表

## 1. 基础完整性

- [ ] 需求来源、需求类型和需求层级已定义。
- [ ] 需求过程覆盖获取、分析、分解、评审、基线、变更和追溯。
- [ ] 需求与架构、验证、发布和配置管理接口明确。

## 2. 输入输出

- [ ] 输入包含客户需求、市场需求、法规约束、安全约束、网络安全约束和项目范围。
- [ ] 输出包含需求清单、需求规格、评审记录、基线、追溯矩阵和变更记录。
- [ ] 需求输出可作为架构设计和验证确认的受控输入。

## 3. 角色职责

- [ ] 产品负责人确认业务目标、优先级和范围。
- [ ] 需求工程师负责需求分析、分解、评审和追溯。
- [ ] 系统、软件、测试、质量、安全和网络安全角色参与相关需求评审。

## 4. 工作产品

- [ ] 需求清单包含编号、描述、来源、优先级、状态和责任人。
- [ ] 需求规格覆盖功能、性能、接口、约束、异常和验收准则。
- [ ] 需求追溯矩阵连接上游需求、设计元素、测试用例和缺陷。
- [ ] 需求变更记录包含原因、影响分析、批准和实施状态。

## 5. 评审与批准

- [ ] 正式需求经过清晰性、完整性、一致性、可行性和可验证性检查。
- [ ] 需求基线经过产品、系统、软件、测试和质量代表批准。
- [ ] 安全和网络安全需求由对应领域负责人确认。

## 6. 配置与变更

- [ ] 需求规格、需求基线、追溯矩阵和评审记录纳入配置管理。
- [ ] 需求变更触发对架构、测试、发布和风险的影响分析。
- [ ] 需求状态与项目计划和发布范围保持一致。

## 7. 度量指标

- [ ] 需求评审通过率可统计。
- [ ] 需求变更率可统计。
- [ ] 需求覆盖率和验证完成率可追踪。
- [ ] 未关闭需求问题数可见。

## 8. 实施证据

- [ ] 需求清单和需求规格有版本记录。
- [ ] 需求评审问题有关闭证据。
- [ ] 需求基线记录可追溯。
- [ ] 需求变更有影响分析和批准记录。

## 9. ASPICE / ISO 33020 关注点

- [ ] 支撑 SYS.2 系统需求分析。
- [ ] 支撑 SWE.1 软件需求分析。
- [ ] 覆盖 PA1.1、PA2.1 和 PA2.2。
- [ ] 为 PA3.1 过程定义提供需求管理规则基础。

## 10. 裁剪规则

- [ ] 概念验证项目裁剪后仍保留需求来源、验收准则和变更记录。
- [ ] 客户交付、安全相关或量产项目未裁剪追溯和基线管理。
- [ ] 裁剪结论有质量负责人确认。
