# 配置管理过程

> 文档编号：MEES-PRO-006
> 版本：v0.2.0
> 状态：已批准
> 所有者：配置管理负责人
> 最后更新：2026-07-14

## 1. 目的

定义配置项识别、基线建立、版本控制、变更控制、状态记录、审计和发布归档的方法，确保工程工作产品可识别、可追溯、可复现、可回退。

## 2. 适用范围

适用于需求、架构、设计、源码、测试用例、测试脚本、构建配置、工具链配置、发布包、评审记录、质量证据和项目管理记录。

## 3. 流程位置

配置管理过程横向支撑产品、项目、需求、系统、软件、测试和发布活动，为 G2、G3、G4、G6 提供需求、设计、实现和发布基线。变更决策及问题闭环由[变更与问题管理过程](07_变更与问题管理过程.md)负责，配置管理负责保存配置状态和版本关系；统一规则见[核心过程总览](00_核心过程总览.md)。

## 4. 输入

| 输入 | 来源 |
|---|---|
| 项目计划、里程碑和发布计划 | 项目管理 |
| 需求、架构、测试和发布工作产品 | 各工程过程 |
| 变更申请、缺陷修复和客户反馈 | 变更管理 / 缺陷管理 |
| 工具链、构建脚本和环境配置 | DevOps / 工程平台 |
| 组织命名、版本和归档规则 | MEES 知识库 |

## 5. 活动

1. 识别项目配置项，定义配置项类型、责任人、存储位置和命名规则。
2. 建立版本控制策略，包括分支、标签、提交、评审和合并规则。
3. 在关键里程碑建立需求基线、设计基线、验证基线和发布基线。
4. 对配置项变更执行影响分析、评审、批准、实施和验证。
5. 记录配置状态，包括版本、基线、变更、缺陷、构建和发布关系。
6. 执行配置审计，检查工作产品完整性、一致性、可追溯性和归档状态。
7. 在发布时冻结、标记、归档交付包和相关证据。

## 6. 输出与工作产品

| 工作产品 | 最小要求 |
|---|---|
| 配置管理计划 | 配置项范围、工具、分支策略、基线规则和职责 |
| 配置项清单 | 编号、名称、类型、责任人、位置、版本和状态 |
| 基线记录 | 基线名称、内容、版本、建立时间、批准人和用途 |
| 变更记录 | 原因、影响分析、批准、实施版本和验证状态 |
| 配置状态报告 | 当前基线、开放变更、发布候选和风险说明 |
| 配置审计记录 | 检查项、发现问题、整改行动和关闭证据 |
| 发布归档清单 | 标签、构建产物、校验信息、测试报告和发布说明 |

## 7. 角色与职责

| 角色 | 职责 |
|---|---|
| 配置管理员 | 维护配置管理计划、配置项清单、基线、标签和归档 |
| 项目经理 | 确认项目级基线计划、变更优先级和发布节奏 |
| 工程负责人 | 确认技术变更影响、合并策略和构建可复现性 |
| 测试负责人 | 确认测试版本、环境配置和验证证据 |
| 发布负责人 | 确认发布候选、交付包、发布标签和归档完整性 |
| 质量负责人 | 执行配置审计，检查证据一致性和过程符合性 |

## 9. 评审与批准

- 配置管理计划需由项目经理、工程负责人、发布负责人和质量负责人评审。
- 每个正式基线需明确范围、版本、批准人和使用场景。
- 影响需求、架构、验证或发布的配置变更需完成影响分析和批准。
- 发布基线需通过配置审计后才能进入正式发布。

## 10. 配置与变更控制

所有正式配置项应纳入受控存储位置。配置项变更必须保留提交记录、评审记录、变更原因、影响分析、批准结论和验证状态。发布后配置项变更必须产生新的版本、补丁或变更记录。

## 11. 度量指标

| 指标 | 数据来源 |
|---|---|
| 配置项受控率 | 配置项清单 / 仓库记录 |
| 基线按期建立率 | 基线记录 / 项目计划 |
| 变更关闭周期 | 变更管理工具 |
| 未批准变更数 | 配置审计记录 |
| 构建可复现率 | CI-CD / 构建记录 |
| 发布归档完整率 | 发布归档清单 |

## 12. 裁剪规则

- 小型探索项目可简化配置管理计划，但必须保留配置项清单、版本记录和发布归档。
- 客户交付、量产、安全相关或网络安全相关项目不得裁剪基线管理、变更记录、配置审计和发布归档。
- 使用外部工具或客户平台时，项目需明确 MEES 配置证据与外部工具记录之间的映射关系。

## 13. 实施证据

- 配置管理计划和配置项清单。
- 分支、标签、提交、评审和合并记录。
- 需求、架构、验证和发布基线记录。
- 变更申请、影响分析、批准和验证证据。
- 配置状态报告、配置审计记录和发布归档清单。
