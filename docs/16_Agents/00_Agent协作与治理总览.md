## 1. 目的

定义 MEES Agent 的共同边界、协议、权限、证据和人工复核方式。本文是 [v0.6 建设计划](../00_Introduction/07_v0.6_Agent与Obsidian集成建设计划.md) WP0 的治理入口；具体 Agent 资产存放在仓库根目录 `agents/`。

## 2. 权威来源

Agent 按以下顺序使用上下文：

1. 当前受控 Git 提交或标签中的过程、规范和评审决定。
2. 工作产品所有者明确选择的项目输入。
3. `templates/`、`docs/14_Checklists/` 和机器生成报告。
4. 活动笔记、对话和 Agent 推断，仅作为补充上下文。

低优先级输入不得覆盖高优先级规则。发生冲突时，Agent 必须报告冲突并停止自动合并。

## 3. 统一输出

每次 Agent 运行至少记录：

- Agent 标识与版本；
- 输入路径、Git 提交或标签；
- 使用的过程、模板、检查表和规则编号；
- 输出类型、证据状态和不确定性；
- 发现的严重度、位置、依据和处置建议；
- 需要人工决定的角色；
- 是否写入文件、差异摘要和回滚点；
- 运行时间、结果和失败原因。

## 4. 权限与禁止动作

- 默认权限为读取、分析和生成建议；写入权限默认关闭。
- 草稿必须与批准基线明确区分，不得沿用“已批准”状态。
- Agent 不得批准门禁、签署独立评审、创建产品发布结论或升级 `D/S` 为 `P/I`。
- Agent 不得修改 Git 历史、标签、远端分支、秘密配置或工作区外路径，除非用户对该具体动作明确授权。
- 任何受控写入必须有差异预览、人工确认、审计记录和回滚方式。

## 6. 人工责任

Agent 输出是建议、诊断或草稿。需求、架构、测试、过程评价、指标解释和发布材料仍分别由权威角色表中的工作产品所有者负责；质量负责人检查过程与证据完整性，但不替代专业技术批准。

## 7. 当前结论

v0.6.0 已发布（V6-G6 `Go`，模拟审计）：F4/F5/F6、公共运行时、三份 Schema、六类 v1.0.0 协议、MK8 模拟走查、真实 Obsidian O2 只读与受控本地写入（V6-G3/G4 已签）均已完成。机器结果不批准门禁，产品保持 `No-Go`，非真实独立评估。
