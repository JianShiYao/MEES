# Obsidian 与 MCP 接入策略

> 文档编号：MEES-LIF-016
> 版本：v0.6.0-dev
> 状态：评审中（V6-G0 已批准）
> 所有者：配置管理员
> 最后更新：2026-07-15

## 1. 目的

规定 MEES 作为 Obsidian Vault 使用时的权威数据路径，以及 MCP 或 Local REST 类适配器的启用条件、权限、秘密、审计和回滚控制。本文落实 [v0.6 建设计划](../00_Introduction/07_v0.6_Agent与Obsidian集成建设计划.md) WP5。

## 2. 当前状态

- 仓库根目录可以直接作为 Obsidian Vault 打开，`.obsidian/` 保存受控基础配置。
- Git、文件系统、MkDocs 和仓库脚本是当前权威读写与验证路径，即 O0/O1 基础可用。
- 已建立不含秘密的 `config/obsidian_integration.json` 和 `scripts/obsidian_adapter.py`；运行审计进入 Git 忽略的 `build/obsidian/`。
- O2/O3 的适配器、降级和临时目录安全测试已实现，但没有真实服务连接或批准范围证据，不得描述为“已连接 Obsidian MCP”。

## 3. 接入层级

| 层级 | 能力 | 当前结论 |
|---|---|---|
| O0 文件系统 | 按受控路径读取、生成补丁并由 Git 管理 | 已具备基础能力 |
| O1 Vault 约定 | Obsidian 浏览、双链、搜索和人工编辑 | 已具备基础能力 |
| O2 MCP/REST 只读 | 获取活动笔记、选择区、搜索结果和 UI 上下文 | REST 兼容适配器候选；真实服务待 V6-G3 |
| O3 MCP/REST 写入 | 创建草稿或修改批准范围内的笔记 | 安全合同测试通过；默认关闭，待 V6-G4 |

## 4. 何时需要 MCP

以下场景才需要 O2/O3：

- 必须知道 Obsidian 当前打开的笔记或选择区；
- 必须调用 Obsidian 搜索、命令或插件能力；
- 用户要求结果立即写入当前笔记，而不是由 Git 补丁更新文件。

批量检查、MkDocs、追溯、指标、Dashboard、CI 和发布不得依赖 MCP。只需读取已知路径时，优先使用 O0。

## 5. 安全与配置

- 服务地址、Token、证书和端口放在本地环境变量或用户私有配置中，不提交仓库。
- 只读与写入使用不同凭据或权限范围；写入默认关闭。
- 明确 Vault 根目录和允许路径，拒绝路径穿越、工作区外访问和未选择附件读取。
- 日志不记录 Token、完整敏感内容或本地绝对项目路径；必要信息使用摘要或脱敏标识。
- 连接失败时降级到 O0，不绕过检查、追溯或门禁。

## 6. 受控写入

O3 每次写入必须：

1. 验证目标处于允许目录且不是已批准基线的直接覆盖操作。
2. 生成补丁或前后差异，显示 Agent、输入和拟修改内容。
3. 获得用户对本次操作的明确确认。
4. 在写入前记录 Git 状态或创建可回滚快照。
5. 原子写入并运行适用检查。
6. 记录结果；失败时恢复到写入前状态并报告未完成项。

## 7. V6-G3 / V6-G4 证据

| 门禁 | 最小证据 |
|---|---|
| V6-G3 | 只读范围清单、秘密检查、活动笔记读取样例、断开连接降级测试、脱敏日志 |
| V6-G4 | 路径白名单、差异预览、显式确认、成功写入、拒绝越权、失败回滚和审计日志 |

## 8. 当前结论

O0/O1 和 O2/O3 适配器候选已实现。真实 O2 连接尚无服务与秘密脱敏证据，V6-G3 保持 `No-Go`；O3 仅完成临时目录测试，仓库默认关闭，须明确写入白名单并单独通过 V6-G4 后才能演练。
