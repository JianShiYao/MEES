## 3. 映射原则

- MEES 过程是组织级工程过程资产，不等同于 ASPICE 单个过程。
- 一个 MEES 过程可以映射多个 ASPICE 过程或支撑过程。
- ISO/IEC 33020 映射关注能力属性，不替代正式评估。
- 当前成熟度只反映 MEES 文档库建设状态，不代表任何项目或组织已经达到对应能力等级。
- 所有标准映射都采用工程解释和证据导向表达，不复制受版权保护的标准原文。

## 4. 能力属性使用方式

| 能力属性 | 在 MEES 中的使用方式 |
|---|---|
| PA1.1 过程执行 | 检查过程是否有目的、输入、活动、输出和实施证据 |
| PA2.1 执行管理 | 检查是否定义计划、责任、资源、监控、风险和调整机制 |
| PA2.2 工作产品管理 | 检查输出物是否定义、评审、配置、变更和归档 |
| PA3.1 过程定义 | 检查组织级标准过程、模板、裁剪规则和接口是否清楚 |
| PA3.2 过程部署 | 检查项目是否按裁剪后的过程执行并留下证据 |
| PA4.1 定量分析 | 检查是否建立量化目标、指标定义和数据来源 |
| PA4.2 定量控制 | 检查是否根据数据识别波动、偏差和控制措施 |
| PA5.1 过程创新 | 检查是否识别改进机会并评价收益 |
| PA5.2 过程优化 | 检查是否部署、验证并固化改进措施 |

## 6. 过程接口矩阵

| 上游过程 | 下游过程 | 关键接口 | 受控证据 |
|---|---|---|---|
| 产品规划过程 | 项目管理、需求管理 | 产品目标、路线图、产品需求和版本目标 | 产品需求基线、G0 记录 |
| 项目管理过程 | 全部专业过程 | 项目范围、裁剪、资源、风险和里程碑 | 项目计划、风险台账、G1 记录 |
| 需求管理过程 | 系统工程过程 | 需求规则、产品需求基线、来源和属性 | 需求清单、基线、评审和追溯记录 |
| 系统工程过程 | 软件工程过程 | 软件分配需求、系统接口、资源和领域约束 | 系统需求、系统架构、接口、G2/G3 记录 |
| 软件工程过程 | 集成与测试过程 | 软件增量、单元结果、构建和集成说明 | 软件基线、单元报告、G4 记录 |
| 集成与测试过程 | 验证确认 | 覆盖、缺陷、分层结论和遗留风险 | 分层测试报告、G5 证据包 |
| 验证确认过程 | 发布管理 | 跨层覆盖、G5 结论、遗留风险和发布适宜性 | G5 记录、发布验证建议 |
| 配置管理过程 | 全部核心过程 | 配置项、基线、版本、审计和归档 | 配置项清单、基线记录、配置状态报告 |
| 变更与问题管理过程 | 全部核心过程 | 问题、影响、决策、实施和验证状态 | 问题/变更台账、CCB 决策和关闭记录 |
| 发布管理过程 | 产品规划、项目管理 | 发布结论、遗留风险、客户反馈和复盘输入 | G6 记录、发布归档、反馈和复盘 |

## 7. 能力等级目标

| 阶段 | 建设目标 | 能力目标 |
|---|---|---|
| v0.1 骨架建立 | 有目录、有入口、有首批过程文档 | 接近 CL1 的文档准备状态 |
| v0.2 核心过程 | 主流程闭环，输入输出和证据清单明确 | 支撑 CL1 过程执行检查 |
| v0.3 标准映射 | 建立 ASPICE、ISO/IEC 33020 和 MEES 的总控映射 | 支撑 CL1 到 CL2 差距分析 |
| v0.4 工程模板 | 为关键工作产品提供可复制模板 | 支撑 PA2.2 工作产品管理 |
| v0.5 自动化与度量 | 建立构建、链接、追溯和质量指标检查 | 支撑 PA2.1 和 PA4.1 的早期数据基础 |
| v1.0 组织级基线 | 形成可评审、可裁剪、可部署的组织过程资产 | 支撑 CL2 到 CL3 建设 |

## 9. 维护规则

- 新增 MEES 过程时，必须同步更新本文的主流程总映射或相关支撑过程映射。
- 修改过程输入、输出、证据或标准映射时，应同步检查本文是否受影响。
- 本文不作为正式 ASPICE 评估结论，只作为 MEES 内部过程建设和自评准备材料。
- 正式评估、客户审核或认证场景应由具备资质的评估人员或领域负责人确认。
