# ASPICE + ISO/IEC 33020 过程映射表

> 本文是 MEES 的原创工程映射，不是 Automotive SPICE、ISO/IEC 33020 或任何标准的官方解释、译文或逐条摘录。

## 1. 目的

建立 MEES 主流程与 ASPICE 过程、ISO/IEC 33020 能力属性、最小实施证据和当前成熟度之间的总控关系，支撑后续过程评审、检查表、模板、度量 Dashboard 和 AI Agent 规则建设。

- [需求管理过程](../01_Main_Process/02_需求管理过程.md)
- [架构设计过程](../01_Main_Process/03_架构设计过程.md)

## 3. 映射原则

- MEES 过程是组织级工程过程资产，不等同于 ASPICE 单个过程。
- 一个 MEES 过程可以映射多个 ASPICE 过程或支撑过程。
- ISO/IEC 33020 映射关注能力属性，不替代正式评估。
- 当前成熟度只反映 MEES 文档库建设状态，不代表任何项目或组织已经达到对应能力等级。
- 所有标准映射都采用工程解释和证据导向表达，不复制受版权保护的标准原文。

## 4. 能力属性使用方式

| 能力属性 | 在 MEES 中的使用方式 |
|---|---|
| PA1.1 过程执行 | 检查过程是否有目的、输入、活动、输出和实施证据 |
| PA2.1 执行管理 | 检查是否定义计划、责任、资源、监控、风险和调整机制 |
| PA2.2 工作产品管理 | 检查输出物是否定义、评审、配置、变更和归档 |
| PA3.1 过程定义 | 检查组织级标准过程、模板、裁剪规则和接口是否清楚 |
| PA3.2 过程部署 | 检查项目是否按裁剪后的过程执行并留下证据 |
| PA4.1 定量分析 | 检查是否建立量化目标、指标定义和数据来源 |
| PA4.2 定量控制 | 检查是否根据数据识别波动、偏差和控制措施 |
| PA5.1 过程创新 | 检查是否识别改进机会并评价收益 |
| PA5.2 过程优化 | 检查是否部署、验证并固化改进措施 |

## 6. 过程接口矩阵

| 上游过程 | 下游过程 | 关键接口 | 受控证据 |
|---|---|---|---|
| 项目管理过程 | 需求管理过程 | 项目范围、发布目标、资源约束、里程碑 | 项目计划、范围基线、需求计划 |
| 需求管理过程 | 架构设计过程 | 需求基线、约束、验收准则、追溯关系 | 需求规格、需求评审记录、需求追溯矩阵 |
| 架构设计过程 | 验证确认过程 | 架构元素、接口、风险、集成策略 | 架构说明、接口说明、架构追溯矩阵 |
| 验证确认过程 | 发布管理过程 | 测试结果、缺陷状态、覆盖率、发布建议 | 测试报告、缺陷清单、发布验证建议 |
| 配置管理过程 | 全部主流程 | 配置项、基线、版本、变更、审计、归档 | 配置项清单、基线记录、配置状态报告、审计记录 |
| 发布管理过程 | 项目管理过程 | 发布结论、遗留风险、客户反馈、复盘输入 | 发布评审记录、发布后反馈记录、项目复盘 |

## 7. 能力等级目标

| 阶段 | 建设目标 | 能力目标 |
|---|---|---|

| v1.0 组织级基线 | 形成可评审、可裁剪、可部署的组织过程资产 | 支撑 CL2 到 CL3 建设 |

## 8. 当前差距

| 差距 | 影响 | 下一步 |
|---|---|---|
| 过程文档仍为草稿 | 只能证明过程定义雏形，不能证明组织批准 | 增加评审记录和批准状态 |
| 缺少工作产品模板 | 过程输出还不能直接标准化复用 | 补需求、架构、测试、发布、配置模板 |
| 缺少专项检查表 | 评审准则仍停留在通用层面 | 补项目、需求、架构、测试、发布和配置检查表 |
| 缺少真实项目证据 | 不能证明 PA1.1 的过程结果已经实现 | 建立 examples 参考项目 |
| 缺少度量数据源 | 暂不能支撑 PA4.1 / PA4.2 | 建立指标定义和采集脚本 |
| 缺少过程裁剪记录 | 暂不能证明 PA3.2 的项目部署 | 建立裁剪申请和批准模板 |

## 9. 维护规则

- 新增 MEES 过程时，必须同步更新本文的主流程总映射或相关支撑过程映射。
- 修改过程输入、输出、证据或标准映射时，应同步检查本文是否受影响。
- 本文不作为正式 ASPICE 评估结论，只作为 MEES 内部过程建设和自评准备材料。
- 正式评估、客户审核或认证场景应由具备资质的评估人员或领域负责人确认。
