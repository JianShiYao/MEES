# 跨标准控制与证据矩阵

> 文档编号：MEES-LIF-008
> 版本：v0.3.0-dev
> 状态：评审中
> 所有者：工程过程负责人
> 最后更新：2026-07-14

## 1. 目的

把 Automotive SPICE、ISO/IEC 33020、ISO 26262、IEC 62443 和 ISO 9001 的共同工程关注点归并到唯一 MEES 控制对象，明确可复用证据、标准专有增量、最终责任和差距，避免重复过程、重复模板和双重批准。

本文使用各专项映射的原创工程解释，不替代标准正文或正式符合性评价。

## 2. 整合原则

1. 同一工作产品只保留一个 MEES 最终责任角色；领域负责人增加属性、准则和批准意见，不取得专业内容的重复所有权。
2. 共同证据优先复用，例如同一需求记录可同时承载来源、ASIL/安全等级、验证准则和质量状态。
3. 标准专有分析保持独立标识，例如 HARA、TARA 和管理评审不能被通用风险台账吞并。
4. 证据复用必须保持版本、适用范围、批准状态和追溯关系，不能以“同名文件”推定充分覆盖。
5. 文档定义 `D`、模拟走查 `S`、真实项目 `P` 和独立评价 `I` 分栏管理。

## 4. 专有控制

| 专有控制 | 不应合并的原因 | 责任角色 | 证据 |
|---|---|---|---|
| HARA、ASIL 和功能安全概念 | 功能安全风险模型、完整性等级和安全目标有专门语义 | 功能安全负责人 | HARA、安全目标、功能安全概念 |
| TARA、区域/通道和 SL-T | 网络威胁、攻击路径和目标安全等级不同于随机/系统性安全失效 | 网络安全负责人 | 安全范围、TARA、区域/通道、SL-T |
| 安全分析与确认措施 | 需要安全专门方法及独立性判定 | 功能安全负责人 | FMEA/FTA 类分析、确认措施记录 |
| 漏洞披露、补丁和 EOL | 涉及保密处理、外部协调和长期产品支持 | 网络安全负责人；产品/发布负责人 | 漏洞记录、公告、补丁、EOL 通知 |
| 内部审核和管理评审 | 评价完整 QMS，不等同于产品技术评审 | 质量负责人；组织最高管理者 | 审核报告、管理评审输出 |

## 5. 唯一责任复核

| 复核项 | 结论 |
|---|---|
| 专业规格 | 系统需求/架构/接口归系统负责人；软件需求/设计/实现归软件负责人 |
| 安全属性 | 功能安全/网络安全负责人定义领域属性和批准意见，不重复拥有专业规格 |
| 测试证据 | 分层执行证据归测试负责人；验证确认过程汇总跨层结论 |
| 配置内容 | 内容技术责任不变；配置管理员只治理标识、基线、状态、审计和归档 |
| 发布 | 领域负责人提供安全/质量建议；发布负责人形成唯一发布决定 |
| 审核 | 质量负责人评价过程和证据，不代替技术责任人批准技术内容 |

复核结论：**未发现跨标准矩阵新增的重复最终责任。** 新增领域角色尚未纳入 v0.2 核心角色权威表，作为 `GAP-V03-FS-001` 和 `GAP-V03-CS-001` 留待 v0.4 修订。

## 6. 证据复用规则

- 同一需求条目可以增加 `safety_class`、`security_level`、`source`、`verification_criteria` 等属性，不为每个标准复制需求正文。
- 同一评审记录可包含多领域准则，但必须记录每个领域的参与人、问题和结论。
- 同一变更单可承载质量、安全和网络安全影响，但各领域“不适用”必须给出理由。
- 同一测试用例可复用执行结果，前提是能分别追溯到普通需求、安全需求和安全威胁处置。
- 同一发布包可复用构建与配置证据，安全偏差、漏洞、回退和客户通知保持独立字段。
