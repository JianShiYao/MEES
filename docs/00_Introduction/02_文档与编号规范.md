# 文档与编号规范

## 1. 目的

规定 MEES 正式文档的命名、编号、结构、状态和链接要求，保证文档可唯一识别、可导航、可评审和可长期维护。根目录 `DOCUMENTATION_GUIDE.md` 是贡献者快速入口，本文是 MkDocs 发布范围内的规范依据。

## 2. 适用范围

适用于 `docs/` 中的正式过程、生命周期总览、评审记录、规范、模板说明、检查表和案例文档。仓库脚本、工具配置和本地 Obsidian 工作区文件不强制采用正式文档编号。

## 3. 文件与目录命名

- 目录使用两位数字前缀保持能力域和导航顺序稳定。
- 中文主题文件使用可辨识名称，不使用“最终版”“最新版”等随时间失效的名称。
- 模板文件以 `_Template` 或“模板”结尾。
- 改名或移动文件时，必须同步更新 Markdown 链接、MkDocs 导航和 README 引用。

## 4. 文档编号

| 编号格式 | 含义 | 使用规则 | 示例 |
|---|---|---|---|
| `MEES-LIF-NNN` | 生命周期、过程架构、跨域总览与策略、建设计划、目录与参考字典 | `NNN` 为三位顺序号 | `MEES-LIF-001` |
| `MEES-PRO-0NN` | 跨域总控过程 | `0NN` 对应总控过程顺序 | `MEES-PRO-002` |
| `MEES-PRO-DNN` | 专业工程过程 | `D` 采用能力域目录的首个有效数字：产品 `1xx`、系统 `3xx`、软件 `4xx`、测试 `5xx` | `MEES-PRO-301` |
| `MEES-MAP-NNN` | 标准映射、跨标准矩阵与差距台账 | `NNN` 为三位顺序号 | `MEES-MAP-001` |
| `MEES-REV-VNN-NNN` | 版本基线评审记录 | `VNN` 表示被评审次版本，末三位为评审顺序号 | `MEES-REV-V02-001` |

编号在仓库内必须唯一。文档拆分、合并、废止或移动后不得把既有编号分配给其他内容；废止文档保留编号和历史状态。没有纳入上述编号域的辅助文档不自行创造新前缀，新增编号类型须先修订本文。

`MEES-MAP-*` 为标准映射文档类，自本次修订起用于新增的标准映射、跨标准矩阵和差距台账。**标识稳定性优先**：v0.3.0 已发布基线中的映射类文档（`MEES-LIF-004`–`009`）保留原有 `LIF` 编号不变，不因本次分类调整而重命名；仅新增此类文档采用 `MEES-MAP-*`。既有编号与新分类的对应关系登记为下述受控技术债台账。

## 5. 文档结构

正式过程文档采用 1–15 节统一结构，至少包含目的、范围、位置、输入、活动、输出、角色、流程、评审、配置与变更、度量、裁剪、证据、标准映射和版本历史。概念或规范类文档可以简化，但必须说明目的、范围、规则和维护方式。

## 6. 状态与批准

| 状态 | 含义 |
|---|---|
| 草稿 | 内容或结构尚未完成，不作为项目约束 |
| 评审中 | 结构完整，正在接受技术和质量评审 |
| 已批准 | 已完成规定批准，可作为组织规范使用 |
| 已废止 | 仅保留历史，不得用于新项目 |

状态变更必须有评审或发布记录支持。“评审中”不能仅因自动检查通过而改为“已批准”。

使用模拟角色完成的过程演练不得只写“已批准”，必须标为“已批准（模拟审计）”，并在评审记录中说明模拟身份不对应真实人员、不构成真实独立批准、评估或认证。

## 7. Markdown 与导航

- 每篇文档只使用一个 H1。
- 内部文档使用相对链接，复杂流程优先使用 Mermaid。
- 新增正式页面必须接入 `mkdocs.yml`；入口页面和 README 按读者路径补充引用。
- 发布前必须执行本地链接检查和 MkDocs 严格构建。
- 标准内容采用原创工程解读和映射，不复制受版权保护的标准原文。

## 8. 维护

文档编号、状态模型或结构要求发生变化时，先更新本文，再同步根目录快速指南、模板、检查表和自动检查规则。
