# MEES 建设路线图

## 1. 目的

明确 MEES 从知识库骨架到可执行研发体系的建设节奏，帮助维护者按优先级补齐过程、模板、检查表、自动化脚本和 AI Agent 能力。

## 2. 建设原则

- 先建立统一结构，再扩展领域内容。
- 先覆盖核心研发闭环，再补充支撑过程。
- 先形成可执行模板，再沉淀案例和自动化。
- 每个新增过程都应包含目的、输入、活动、输出、角色、度量、证据和标准映射。
- 所有内容以原创工程解读和实施指导为主，不复制受版权保护的标准原文。

## 3. 阶段规划

| 阶段            | 目标                     | 主要交付物                                                                | 状态  |
| ------------- | ---------------------- | -------------------------------------------------------------------- | --- |
| v0.1 骨架建立     | 建立目录、规范和最小可阅读入口        | 总览、文档规范、版本规范、ISO/IEC 33020 解读、模板、检查表、MkDocs 配置                       | 进行中 |
| v0.2 核心过程     | 补齐 Agile + Mini V 的主流程 | 项目管理、需求、架构、设计、实现、测试、发布过程文档                                           | 已启动 |
| v0.3 标准映射     | 建立过程与标准之间的工程映射         | ASPICE、ISO/IEC 33020、ISO 26262、IEC 62443、ISO 9001 映射表                | 计划中 |
| v0.4 工程模板     | 让项目团队可以直接复制使用          | 需求、架构、测试、风险、评审、发布、问题闭环模板                                             | 计划中 |
| v0.5 自动化与度量   | 建立基础检查和质量数据入口          | 链接检查、追溯矩阵、质量指标汇总、Dashboard 原型                                        | 计划中 |
| v0.6 Agent 协作 | 让 AI 能按规则辅助工程活动        | Requirement Agent、Review Agent、Test Agent、ASPICE Agent、Metrics Agent | 计划中 |
| v1.0 组织级基线    | 形成可评审、可裁剪、可落地的组织过程资产   | 已评审过程库、模板库、检查表库、案例项目、发布说明                                            | 计划中 |

## 4. 优先补齐清单

1. 修复文件名、导航和入口链接的一致性。
2. 编写项目管理、需求管理、架构设计、验证确认和发布管理的首批过程文档。
3. 建立 ASPICE 与 ISO/IEC 33020 的过程能力映射框架。
4. 扩展文档模板为可直接落地的需求、设计、测试和评审模板。
5. 增加 Markdown 链接检查、MkDocs 构建检查和 Mermaid 检查脚本。
6. 建立参考项目，用一个嵌入式控制器案例贯穿需求、架构、测试、发布和现场问题闭环。

## 5. 近期里程碑

| 里程碑 | 验收标准 |
|---|---|
| M1 文档入口可用 | README、MkDocs 导航和实际文件路径一致，核心页面可打开 |
| M2 主流程闭环 | 从产品规划到发布验证至少有一条完整过程链 |
| M3 模板可复制 | 新项目可直接复制模板并产生基础工程证据 |
| M4 检查可自动化 | 本地和 CI 能检查文档构建、链接和基础格式 |
| M5 案例可演示 | examples 中有一套最小参考项目贯穿核心工程活动 |

## 6. 维护节奏

- 每个次版本聚焦一个能力域或一条主流程。
- 每次发布前检查导航、链接、模板、检查表和版本历史。
- 涉及强制过程要求、标准映射或质量门禁的变更，需要领域负责人和质量负责人评审。
