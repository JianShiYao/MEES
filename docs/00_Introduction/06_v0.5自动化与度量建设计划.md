# v0.5.0 自动化与度量建设计划

> 文档编号：MEES-LIF-012
> 版本：v0.5.0-dev
> 状态：草稿（V5-G0 准备）

## 1. 目的

把 v0.4.0 已批准的过程字段、工程模板和证据边界转化为可重复执行的自动检查、追溯生成、指标采集和可视化能力，完成路线图里程碑 M5“检查可自动化”。

v0.5 同时建设两类试点方法：嵌入式目标级验证方法，以及储能系统功能安全方法。两者用于检验自动化和度量是否能承载真实工程语义，不改变 v0.5 以自动化与度量为主线的范围。

| 输入 | 用途 |

### 2.1 准入判断

v0.5 的规划前置条件已满足：V4-G5 为 `Go`，18 项 v0.4 模板字段已冻结，模板层开放严重和主要问题为 0，现有链接、模板和 MkDocs 检查可重复执行。

仍保留以下边界：MK8 实例主要是 `S` 类模拟证据；自动生成的矩阵、指标和 Dashboard 不得升级其证据等级，也不得被解释为产品批准、标准符合性、认证、独立评估或过程能力等级。

## 3. 建设目标

1. 建立一个跨 Windows 与 Linux 可重复执行的 MEES 检查入口。
2. 把文档、模板、编号、导航、Mermaid、占位符和追溯规则转化为带规则编号的诊断。
3. 从受控 Markdown 实例生成需求到验证、问题到发布的双向追溯矩阵。
4. 冻结首批质量指标的数据源、公式、责任、频率、阈值和证据等级。
5. 生成静态 Dashboard，展示基线健康度、追溯完整性、验证状态、问题和门禁就绪度。
6. 建立嵌入式目标级验证方法，区分主机、SIL/PIL、目标硬件和 HIL 证据。
7. 建立 ESS 功能安全管理、危害/SIL 和 UL 9540A 证据接口的最小方法闭环。
8. 使用 MK8/ESS 模拟数据完成端到端运行，确保缺失证据会得到明确失败或 `No-Go`，而不是生成虚假绿灯。

## 4. 非目标

- 不在 v0.5 建设完整 ALM、数据仓库、实时监控平台或多用户权限系统。
- 不直接解析或上传被外层仓库忽略的 MK8 原始源代码、客户数据或授权不明制品。
- 不以自动化结果替代需求、架构、安全、测试、质量或发布角色的专业批准。
- 不宣称 Automotive SPICE、ISO/IEC 33020、IEC 61508、UL、ISO 26262、IEC 62443 或 ISO 9001 符合性、认证或能力等级。
- 不在 v0.5 完成真实 UL 9540A、HIL、目标板、故障注入或独立功能安全评估；仅定义接口并使用模拟证据走查。
- 不在 v0.5 接入可修改仓库内容的 Agent 或 Obsidian MCP 自动操作；该能力属于 v0.6 治理范围。

## 5. 自动化合同

### 5.1 统一入口

计划建立 `scripts/check_mees.py` 作为统一入口，编排现有和新增检查。各检查器仍可独立运行，统一入口负责聚合结果、退出码和报告。

| 项目 | 约定 |
|---|---|
| 输入 | 默认读取 Git 受控文件；可通过显式路径检查干净导出或项目实例 |
| 输出 | 人类可读控制台结果，以及稳定 JSON 报告 |
| 诊断字段 | `rule_id`、`severity`、`path`、`line`、`message`、`evidence_state`、`remediation` |
| 严重度 | `error` 阻断、`warning` 需处置、`info` 提示 |
| 退出码 | `0` 通过、`1` 发现规则失败、`2` 工具或输入错误 |
| 路径 | 报告统一使用仓库相对 POSIX 路径，运行时兼容 Windows 与 Linux |
| 生成目录 | 默认进入被 Git 忽略的 `build/` 或 `site/`，不污染受控源文件 |

### 5.2 首批规则域

| 规则域 | 最小检查 |
|---|---|
| 链接与导航 | 本地链接目标存在；MkDocs 导航文件存在；新增正式页未形成孤立入口 |
| 编号与元数据 | 文档、评审、模板编号唯一；版本、状态、所有者和证据状态字段有效 |
| 模板合同 | v0.4 冻结数量、负责人、章节和版本状态一致；v0.5 增量单独计数 |
| 占位符 | 已批准基线和项目实例不得残留未处置的 `[填写：...]`；`TBD` 必须有责任与关闭条件 |
| Mermaid | 围栏成对、图类型受支持；在 CI 中执行 Mermaid 语法渲染验证 |
| 追溯 | 标识格式、引用目标、允许边类型、孤立对象、重复关系和断链检查 |
| 证据边界 | `D/S/P/I` 取值有效；模拟输入不得自动升级为 `P/I` |
| 基线一致性 | 版本、状态、候选提交、发布决定和标签之间的声明不互相矛盾 |

禁止用正则表达式拼接复杂 Markdown 表格语义；追溯和指标采集优先使用 Markdown 解析器或结构化中间模型。

## 6. 首批追溯模型

### 6.1 对象

首批支持 `PGO`、`PRD`、`SYS-REQ`、`SYS-ARC`、`EXT-HW`、`SWE-REQ`、`SWE-ARC`、`SWE-DD`、`SWU`、`TST`、`CHG`、`BL`、`REL`、`HAZ`、`SIL` 和 `SAFE-FN` 对象。

### 6.2 输出

- `build/traceability/traceability.json`：机器读取的对象、关系和诊断。
- `build/traceability/traceability.csv`：便于筛选和人工抽查的关系表。
- `build/traceability/traceability.md`：MkDocs 可读的矩阵摘要。
- 正向与反向查询：需求到验证、缺陷/变更到发布、安全危害到安全功能与验证。

生成器只能汇总源记录，不得创造缺失关系；断链必须进入诊断和门禁就绪度，而不是自动补齐。

## 7. 首批指标

每个指标必须在指标字典中定义标识、目的、公式、数据源、粒度、责任角色、采集频率、阈值、异常处置、证据状态和限制。

| 指标标识 | 指标 | 首批用途 |
|---|---|---|
| `MET-DOC-001` | 文档检查通过率 | 监控链接、构建、编号和模板规则健康度 |
| `MET-TPL-001` | 模板实例完整率 | 识别必填字段、批准和归档缺口 |
| `MET-TRC-001` | 双向追溯完整率 | 识别孤立需求、设计、测试和发布对象 |
| `MET-VNV-001` | 需求验证覆盖率 | 统计有已定义验证关系的需求比例 |
| `MET-VNV-002` | 测试执行率 | 区分计划、执行和未执行测试 |
| `MET-VNV-003` | 测试通过率 | 仅对已执行且证据有效的测试计算 |
| `MET-ISS-001` | 开放问题严重度分布 | 支撑升级、门禁和发布风险判断 |
| `MET-REL-001` | 发布门禁就绪度 | 汇总 G0–G6 必须证据，不替代签署 |
| `MET-QA-001` | 不符合项逾期率 | 支撑 QA 监督和 CAPA 升级 |
| `MET-SUP-001` | SBOM/供应方证据完整率 | 识别组件、许可、漏洞和外部硬件证据缺口 |

当分母为 0、输入证据为 `S` 或数据不完整时，Dashboard 必须显示“不适用/无充分证据”，不得显示误导性的 100%。

## 8. Dashboard 最小范围

Dashboard 采用脚本生成的静态 HTML 和 JSON 数据，不引入服务端或数据库。首版至少包含：

- 基线身份、生成时间、源提交和证据状态分布。
- 自动检查总览及按严重度、规则域和文件的诊断。
- 追溯完整率、孤立对象及需求到验证矩阵入口。
- 测试计划/执行/通过状态，并突出“0 项执行”场景。
- 开放问题、QA 不符合项、CAPA 和发布门禁就绪度。
- SBOM、供应方及 `EXT-HW` 证据缺口。
- MK8/ESS 模拟数据的醒目标识，避免与真实项目指标混淆。

视觉实现以工程扫描和问题定位为主，不建设营销式首页。Dashboard 的每个汇总值必须能回到指标定义和源记录。

## 9. 嵌入式工程方法与目标级验证

v0.5 建立一篇受控方法文档和一份检查表，至少覆盖：

1. 编译器、链接器、启动代码、内存布局、构建变体和制品身份。
2. 编码规则、静态分析、单元验证、覆盖率和偏差批准。
3. 主机单元测试、SIL、PIL、目标硬件测试和 HIL 的目的及证据不可互换性。
4. 接口、时序、资源、并发、中断、看门狗、启动/升级和故障恢复验证。
5. 故障注入、边界值、降级/安全状态和诊断覆盖验证。
6. 测试环境、桩件、测量设备、工具版本和环境配置的可复现性。
7. 需求、架构、代码单元、测试、缺陷、构建和发布之间的追溯。

MK8 试点只形成模拟验证计划和证据合同；没有目标板或 HIL 结果时，相关验证状态必须保持“未执行”。

## 10. ESS 功能安全方法试点

基于已建立的储能安全映射和 `TPL-V05-001`，v0.5 最小建设：

- 储能功能安全管理过程：范围、角色、独立性、裁剪、修改控制和安全生命周期接口。
- 完成危害分析与 SIL 确定模板，保持 IEC 61508 SIL 与汽车 ASIL 术语隔离。
- 建立 UL 9540A 热失控测试计划/报告证据接口模板；测试结果作为危害表征输入，不替代风险分析或认证。
- 定义电芯/电池 `EXT-HW` 型式试验、验收、变更通知和供应方证据接口。
- 使用脱敏模拟 ESS 场景形成 `HAZ → SIL → SAFE-FN → SYS/SWE-REQ → TST → REL` 走查；缺真实试验时发布保持 `No-Go`。

受控标准正本版本、适用条款和项目适用性仍需领域负责人确认；仓库仅保存原创工程解读和接口，不复制受版权保护的标准原文。

## 13. M5 验收标准

- [ ] 一个命令可执行全部仓库检查，并提供稳定 JSON 与控制台结果。
- [ ] 链接、MkDocs、模板、编号、导航、Mermaid、占位符、追溯和证据状态均有规则编号。
- [ ] 每个失败能定位到规则、文件、行和处置建议。
- [ ] 自动检查在 Windows 本地和 GitHub Actions Linux 环境结论一致。
- [ ] 追溯生成器输出 JSON、CSV、Markdown，并支持至少三条双向查询链。
- [ ] 首批至少 10 项指标完成口径评审；无数据或模拟证据不会产生虚假达标值。
- [ ] 静态 Dashboard 展示检查、追溯、验证、问题、发布和供应方证据状态。
- [ ] 嵌入式目标级验证方法覆盖主机、SIL/PIL、目标硬件与 HIL 的边界和证据。
- [ ] ESS 试点覆盖 FSM、危害/SIL、UL 9540A 接口及一条模拟安全追溯链。
- [ ] MK8/ESS 试点在缺真实测试和批准时保持 `No-Go`。
- [ ] 自动化脚本具备针对成功、规则失败、无数据、格式错误和断链的测试。
- [ ] 候选提交在干净导出目录复现全部检查和 Dashboard 生成。
- [ ] 人工评审开放严重和主要问题为 0，一般问题有责任人、目标版本和关闭判据。

## 14. 进入 v0.6 的门禁

只有满足以下条件，才允许 Agent 或 Obsidian MCP 消费 v0.5 输出：

1. V5-G6 通过，v0.5.0 自动化输出合同和规则编号已冻结。
2. JSON 报告、追溯图和指标字典具有版本字段及兼容性策略。
3. 至少一个项目实例证明诊断和指标能被人工复核，且证据等级不被自动升级。
4. Agent 首期只能读取受控输出并提出建议，不直接批准门禁、修改证据状态或发布产品。
5. Obsidian MCP 的读取范围、写入权限、敏感信息、审计日志和失败回滚策略已评审。
6. 自动化和方法层开放严重、主要问题为 0。

## 15. 优先级与实施顺序

1. 先执行 WP0，批准 V5-G0 后冻结数据合同。
2. WP1 与 WP2 先使用 v0.4 MK8 模拟实例开发，保持小数据集可人工核对。
3. WP3 在追溯中间模型稳定后冻结指标，避免 Dashboard 反向驱动字段漂移。
4. WP4 先生成静态报告，再接入 CI；不先建设实时服务。
5. WP5 与 WP6 使用同一追溯和指标合同验证专业方法，不另建私有数据模型。
6. WP7 执行干净导出复现、人工评审和 V5-G6 决策。

P0 范围是 WP0–WP4 的基础自动化、目标级验证证据边界，以及 ESS 的 FSM/危害/SIL/UL 9540A 最小链。趋势数据库、SARIF/JUnit 扩展、复杂交互 Dashboard 和真实硬件接入属于 P1，可在不影响 M5 的条件下递延。

| 风险 | 控制措施 |
|---|---|
| Markdown 自由文本难以可靠解析 | 冻结受控表格字段，使用解析器和结构化中间模型；格式错误明确失败 |
| 自动化修改工作产品语义 | 工具默认只读，只生成报告；修复建议不得自动批准或升级证据 |
| 指标驱动错误行为 | 每项指标声明目的、限制和配套审阅，不单独用于绩效评价 |
| 模拟数据显示虚假绿灯 | Dashboard 强制显示证据状态；无数据/未执行独立于失败和通过 |
| CI 与 Windows 本地结果漂移 | 固定依赖版本、路径规范和时区；建立跨平台测试样例 |
| Mermaid 工具链引入 Node 依赖 | 把渲染校验隔离为受控 CI 步骤，并保留 Python 围栏基础检查 |
| ESS 范围挤压 M5 主线 | ESS 只建设最小方法与模拟链；真实试验、认证和完整安全案例递延 |
| 工具接触客户源码或敏感数据 | 默认仅扫描 Git 受控 MEES 资产；外部输入需显式路径和脱敏批准 |
