# 版本规范

## 1. 目的

规定 MEES 仓库版本、文档版本、候选提交和发布标签的关系，使评审对象与自动验证证据可复现。根目录 `VERSIONING.md` 是贡献者快速入口，本文是 MkDocs 发布范围内的规范依据。版本发布指**知识库/方法基线**发布而非产品发布，产品状态与证据边界以 [边界与证据声明](09_边界与证据声明.md) 为准。

## 2. 版本格式

MEES 使用语义化版本 `主版本.次版本.修订版本`：

- 主版本：体系结构、兼容性或治理方式发生重大变化。
- 次版本：增加新的能力域、过程、模板或案例。
- 修订版本：错误修正、澄清和不影响总体结构的小幅改进。

## 3. 文档版本

正式文档必须维护版本历史。批准后的内容修正至少提升修订版本；过程责任、输入输出、门禁或强制要求变化时提升次版本。候选阶段可以保持目标版本号，但状态必须为“草稿”或“评审中”，不得以版本号代替批准状态。

## 4. 候选版本与提交

- 开发候选使用 `vX.Y.Z-dev` 表示，例如 `v0.2.0-dev`。
- 基线评审必须记录候选内容的完整 Git 提交号。
- 自动验证应在该提交的干净导出目录或等效干净检出中执行，并记录命令、结果和可复现数字。
- 整改改变候选内容后，原提交保留为历史证据；必须冻结新候选提交并重新执行检查。

## 5. 发布标签

正式发布标签采用 `vX.Y.Z`，例如 `v0.1.0`、`v0.2.0` 和 `v1.0.0`。只有在开放严重和主要问题关闭、规定角色签署最终 `Go`、严格构建通过且变更记录更新后，才可创建目标版本标签。

Conditional Go 允许继续准备后续工作，但不允许创建正式标签、将过程状态改为“已批准”或对外声明能力等级达标。

## 6. 发布记录

每次正式发布至少更新：

- `CHANGELOG.md` 的版本条目和发布日期；
- README 当前版本和状态；
- 建设路线图和待办状态；
- 对应基线评审记录、候选提交和最终签署；
- MkDocs 导航、链接检查和严格构建证据。

## 7. 维护

版本策略、候选命名或发布门禁变化时，先更新本文，再同步根目录快速指南、CI 发布流程和基线评审模板。
