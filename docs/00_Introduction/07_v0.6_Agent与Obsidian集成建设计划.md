## 1. 目的

在 v0.5.1 可检查、可追溯、可度量的内部基线上，建立受治理的 AI Agent 协作能力和 Obsidian 接入方式，使 Agent 能读取受控过程、模板、检查表、案例和机器报告，生成可复核建议或草稿，并通过 BMS 参考项目证明这些能力不会绕过角色责任、证据等级和发布门禁。

v0.6 的核心不是增加聊天入口，而是把 Agent 的输入、输出、权限、证据、诊断、人工复核和失败回滚定义为工程合同。

## 3. 建设目标

1. 关闭 v0.5 遗留 F4/F5/F6，使 Agent 消费的是可统一执行、追溯告警受控且 Mermaid 可真实验证的输入。
2. 建立 Requirement、Architecture、Review、Test、ASPICE、Metrics 六类 Agent 的统一协议和最小可运行切片。
3. 让 Agent 的每项结论可追溯到受控输入、规则、模板、检查表或机器报告。
4. 建立文件系统优先的 Obsidian 集成；只有需要当前活动笔记或 UI 状态时才启用 MCP/Local REST 适配层。
5. 先完成只读检索和建议，再开放限定目录、限定文件和显式确认下的受控写入。
6. 建立审计日志、差异预览、失败回滚和敏感信息边界。
7. 使用 BMS 参考项目跑通需求、架构、测试、评审、追溯、度量和发布建议链。
8. 形成 M6 可演示、可复现、可人工评审的 Agent 协作基线。

## 4. 非目标

- Agent 不替代产品、项目、系统、软件、测试、质量、配置或发布负责人的最终责任。
- Agent 不自动批准 G0–G6、V6-G0–G6、功能安全确认、独立评估或产品发布。
- Agent 不把草稿、模拟数据或推断结果升级为 `P/I` 证据。
- v0.6 不承诺任意 Obsidian 插件、任意 MCP 客户端或云端模型均可无配置运行。
- 不允许 Agent 无差异预览地批量修改受控文档、模板、证据状态、标签或远端分支。
- 不在 v0.6 声明组织能力等级、标准符合性、认证或真实 BMS 产品批准。

## 5. Agent 治理合同

### 5.1 权限等级

| 等级 | 能力 | v0.6 规则 |
|---|---|---|
| R0 | 读取公开受控文件 | 默认允许，必须记录输入路径和版本 |
| R1 | 读取本地项目或活动笔记 | 仅在用户明确选择范围后允许，不进入发布证据 |
| A1 | 分析、诊断、查询和建议 | 默认 Agent 工作模式，输出必须标注依据与不确定性 |
| D1 | 生成新草稿或补丁 | 必须使用模板、显示差异并等待人工复核 |
| W1 | 写入已批准范围 | 仅在 V6-G4 后开放；限定路径、操作和回滚点 |
| G0 | 门禁批准、证据升级、标签和发布 | Agent 禁止执行最终决定，只能准备材料 |

### 5.2 统一协议字段

每个 Agent 必须定义：标识与版本、目的、适用范围、输入模式、受控依据、输出模式、权限等级、禁止动作、证据状态、诊断格式、人工复核角色、失败行为、审计字段、测试样例和版本历史。

所有结构化输出优先使用 JSON Schema 或等价机器可校验格式；自然语言报告必须能回链到结构化结果，不得成为唯一证据源。

### 5.3 六类 Agent

| Agent | 主要输入 | 主要输出 | 最终人工责任 |
|---|---|---|---|
| Requirement Agent | 产品目标、需求模板、变更和追溯报告 | 需求草稿、歧义/缺口诊断、追溯建议 | 需求工程负责人或专业负责人 |
| Architecture Agent | 已批准需求、接口和架构模板 | 架构候选、接口一致性诊断、分配建议 | 系统负责人或软件负责人 |
| Review Agent | 工作产品、检查表、规则报告 | 分级发现、依据、处置建议和复核清单 | 工作产品所有者与质量负责人 |
| Test Agent | 需求、架构、验证方法和历史缺陷 | 分层测试候选、覆盖缺口和执行前检查 | 测试负责人 |
| ASPICE Agent | 过程文档、映射、证据索引和评审记录 | 过程结果/实践覆盖诊断、差距建议 | 工程过程负责人和质量负责人 |
| Metrics Agent | 检查、追溯、指标和项目快照 | 指标解释、异常诊断和改进行动草稿 | 质量负责人和指标所有者 |

## 6. Obsidian 接入策略

### 6.1 分层接入

| 层级 | 接入方式 | 用途 | 门禁 |
|---|---|---|---|
| O0 | Git/文件系统直接读写 | CI、批处理、稳定路径和版本化内容 | V6-G1 |
| O1 | Obsidian Vault 约定与链接 | 人工浏览、双链、模板和知识组织 | V6-G2 |
| O2 | MCP 或 Local REST 只读适配 | 当前活动笔记、选择区、搜索和 UI 上下文 | V6-G3 |
| O3 | MCP 或 Local REST 受控写入 | 创建草稿、插入建议、更新限定记录 | V6-G4 |

O0 是所有自动化与 CI 的权威路径。O2/O3 是本地交互适配层，不得成为 MkDocs、检查、追溯或发布的必要依赖。

### 6.2 接入前置控制

- 明确 Vault 根目录、允许读取目录、允许写入目录和禁止路径。
- Token、端口和证书只存本地安全配置，不进入仓库、日志或示例。
- 只读操作与写操作使用不同权限；写操作默认关闭。
- 每次写入先生成补丁或差异预览，记录操作者、Agent、时间、输入、输出和结果。
- 写入前创建可回滚点；失败时不得留下半完成状态。
- 活动笔记、剪贴板、附件和本地项目内容默认视为非发布证据。
- 未安装 Obsidian、插件或 MCP 时，仓库检查和 Agent 离线样例仍须可运行。

## 7. v0.5 遗留项关闭

| 遗留项 | v0.6 工作 | 关闭判据 |
|---|---|---|

## 8. BMS 参考项目

v0.6 使用脱敏 BMS 示例建立一条最小可演示链：

`产品目标 → 系统/软件需求 → 架构与接口 → 测试设计 → 评审发现 → 变更/问题 → 构建与发布建议 → 指标反馈`

每个环节至少由一个 Agent 读取受控输入、生成可审查输出，并由对应人工角色接受、修改或拒绝。项目发布仍保持 `No-Go`，直到真实执行证据和责任人批准齐备。

## 12. M6 验收标准

- [ ] 一个命令完成文档、模板、追溯、指标、Dashboard、Mermaid 和 MkDocs 检查，并输出稳定控制台与 JSON 结果。
- [ ] 单次追溯报告至少提供三条可验证的正反查询。
- [ ] MK8 15 项、ESS 4 项历史 warning 均有关闭或逐项接受记录。
- [ ] 全部 Mermaid 围栏经过真实解析/渲染检查，错误可定位。
- [ ] 六类 Agent 均有版本化协议、结构化输入输出、权限、禁止动作、人工责任和自动测试。
- [ ] Agent 输出能回链到受控输入，不自动批准门禁、升级证据或发布产品。
- [ ] Obsidian 未安装或 MCP 不可用时，O0 文件系统路径和 CI 保持可用。
- [ ] O2 只读接入不泄露 Token、敏感路径或未选择内容。
- [ ] O3 写入只作用于批准范围，并具备差异预览、显式确认、审计日志和回滚证据。
- [ ] BMS 参考项目跑通需求到发布建议及指标反馈，真实执行缺失时保持 No-Go。
- [ ] Windows 本地与 GitHub Linux 结论一致，候选提交在干净导出目录复现。
- [ ] 人工评审开放严重和主要问题为 0；其他问题有责任人、目标版本和关闭判据。

## 13. 进入后续版本的门禁

## 15. 风险与控制

| 风险 | 控制 |
|---|---|
| Agent 生成内容看似完整但依据不足 | 强制输入引用、证据状态、不确定性和人工复核 |
| Agent 越权批准或修改基线 | G0 禁止动作、路径白名单、差异预览和分支保护 |
| Obsidian 插件或 MCP 成为单点依赖 | O0 文件系统为权威路径，O2/O3 可关闭和降级 |
| Token、客户数据或活动笔记泄露 | 本地秘密、最小读取范围、日志脱敏和禁止提交检查 |
| 自动写入破坏人工内容 | 写前快照、补丁预览、显式确认、原子写入和回滚测试 |
| 模拟案例被误当真实项目证据 | 固定 `S` 标识，发布与产品门禁保持 No-Go |
| 多 Agent 结论互相覆盖 | 唯一工作产品所有者、建议合并记录和冲突升级规则 |
